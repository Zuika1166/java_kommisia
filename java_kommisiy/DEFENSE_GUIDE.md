# Гайд по защите

## Краткое описание проекта

Это консольное приложение для сервисного центра. Оно позволяет:

- вести склад запчастей;
- оформлять заказы на ремонт;
- искать запчасти по названию;
- фильтровать заказы по статусу.

Проект написан на Java 17 с использованием JDBC и SQLite. Архитектура построена по DAO Pattern.

## Что говорить преподавателю

Можно рассказать так:

> Я реализовал консольное приложение для учета запчастей и заказов сервисного центра. Для доступа к базе данных использовал JDBC без Hibernate. Архитектура построена по DAO Pattern: модели описывают сущности, DAO-интерфейсы задают CRUD-операции, JDBC-реализации DAO работают с SQL, сервисный слой содержит бизнес-логику, а ConsoleMenu отвечает за работу с пользователем. Ключевой процесс оформления заказа реализован в транзакции.

## Структура проекта

### `App`

Точка входа.

Что делает:

- запускает инициализацию базы;
- создает DAO;
- создает сервисы;
- запускает меню.

### `config`

#### `DatabaseConfig`

Читает настройки подключения из `db.properties`.

#### `ConnectionFactory`

Создает JDBC-соединение.

#### `DatabaseInitializer`

Выполняет `schema.sql` и `data.sql` при запуске.

### `model`

#### `Part`

Модель запчасти:

- `id`
- `name`
- `manufacturer`
- `price`
- `quantity`

#### `Order`

Модель заказа:

- `id`
- `customerName`
- `deviceModel`
- `partId`
- `quantity`
- `totalPrice`
- `status`
- `createdAt`

#### `OrderStatus`

Перечисление статусов:

- `NEW`
- `IN_PROGRESS`
- `COMPLETED`
- `CANCELLED`

### `dao`

#### `PartDao`

CRUD для запчастей и поиск по названию.

#### `OrderDao`

CRUD для заказов и фильтрация по статусу.

### `dao.impl`

#### `PartDaoImpl`

Работает с таблицей `parts` через JDBC.

Важно:

- используется `PreparedStatement`;
- поиск сделан через `LIKE`.

#### `OrderDaoImpl`

Работает с таблицей `repair_orders`.

### `service`

#### `PartService`

Проверяет корректность данных запчастей и вызывает DAO.

#### `OrderService`

Самый важный класс.

Что делает:

- проверяет наличие запчасти;
- списывает запчасть со склада;
- рассчитывает сумму;
- создает заказ;
- использует транзакцию.

## Как объяснить транзакцию

В `OrderService.placeOrder()`:

1. открывается соединение;
2. вызывается `setAutoCommit(false)`;
3. считывается запчасть;
4. проверяется остаток;
5. уменьшается количество;
6. создается заказ;
7. вызывается `commit()`;
8. если произошла ошибка, вызывается `rollback()`.

Смысл транзакции:

- заказ и списание со склада должны пройти вместе;
- нельзя допустить частичное выполнение операции.

## Поток данных

Цепочка работы такая:

`ConsoleMenu -> Service -> DAO -> Database`

Пример оформления заказа:

1. пользователь вводит данные в меню;
2. `ConsoleMenu` передает их в `OrderService`;
3. `OrderService` делает проверки и запускает транзакцию;
4. DAO выполняют SQL;
5. результат возвращается обратно в меню.

## Частые вопросы и ответы

### Почему DAO Pattern?

Чтобы разделить ответственность между слоями и не смешивать SQL, бизнес-логику и пользовательский интерфейс.

### Почему JDBC, а не Hibernate?

Потому что по условию задания нужно использовать JDBC без ORM.

### Почему `PreparedStatement`?

Он безопаснее и удобнее для параметризованных SQL-запросов.

### Зачем нужен сервисный слой?

В нем находятся проверки, расчет суммы и транзакционная логика, а не только доступ к БД.

### Зачем нужна транзакция?

Чтобы списание запчасти и создание заказа происходили как одна атомарная операция.
